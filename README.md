# 외환차손익 · 외화환산손익 검증 자동화 도구

회사가 제시한 **분개장 / 명세서(외화자산부채명세서) / 계정별원장**을 입력받아, 기말에 계상된
외환차손익(실현손익)과 외화환산손익(미실현손익)이 적정한지 자동으로 검증하는 감사 절차 자동화 도구입니다.

파트타임 감사 실무 중 직접 수행했던 절차 — *"표본을 뽑아 증빙(은행 외환거래확인서 등)을 일일이
대조하며 적용환율과 외환차손익을 검증하는 작업"* — 을 자동화하는 것에서 출발했습니다.

## 왜 이렇게 설계했는가

외화환산손익(기말)과 외환차손익(결제 시점)은 검증 방식이 근본적으로 다릅니다.

- **외화환산손익**: 결산일 하나의 공식 환율(한국수출입은행 매매기준율)로 전사 일괄 재평가하므로,
  "정답"이 하나뿐입니다. → **전수 자동 검증**이 가능합니다.
- **외환차손익**: 실제로 결제된 은행·계좌 조건에 따라 전신환매매율, 우대환율 등 다른 환율이
  적용됩니다. "정답 환율"이 존재하지 않고, 증빙을 봐야만 실제 적용 환율을 알 수 있습니다.
  → 전수조사 대신, **이상치를 자동으로 골라내고(스크리닝) 그 건에 대해서만 증빙을 대조**하는
  2단계 구조로 설계했습니다.

```
분개장 + 명세서 + 계정별원장
        │
        ▼
[0] 거래처별 연간 요약 + 월별 환율 추이 (분석적 절차)
    실제 감사에서는 거래를 한 줄씩 보기 전에 거래처별 연간 손익 요약과
    월별 환율 추이부터 훑어보고, 숫자가 튀는 거래처/시점만 드릴다운한다.
        │
        ▼
[A] 외환차손익 (결제 건)                    [B] 외화환산손익 (기말 미결제 건)
        │                                           │
  1단계: 전수 스크리닝                        결산일 공식환율로
  회사 적용환율(내재) vs                      전수 자동 재계산·검증
  공식 매매기준율(수출입은행 API)                    │
  괴리율 5% 초과 → 이상치 플래그               (재평가 누락 건도 명세서 대조로 자동 탐지)
        │
  보조1: 합계 중요성(AMPT) 검증
  개별 통과 건도 재계산-계상액 차이를 전부 합산해
  허용가능 오류금액 초과 여부 확인
        │
  보조2: 결제일-환율기준일 불일치 탐지
  "전월말 환율을 잘못 썼다" 같이 크기는 작아도
  실제로는 다른 날짜 환율과 정확히 일치하는 경우를 탐지
        │
  2단계: 이상치 건만 증빙(은행 외환거래확인서 등 이미지)을
  Claude Vision으로 읽어 실제 적용환율 추출 → 최종 대사

[C] 계정별원장 대사
    분개장에서 집계한 외환차익/차손/외화환산이익/손실 계정 합계가
    총계정원장 잔액과 일치하는지 확인 (수기 조정분 누락 등 완전성 이슈 탐지)
```

## 처리 가능한 실무 케이스

- **분할결제**: 한 거래가 여러 날짜에 나눠 결제되는 경우, 결제 건별로 정확히 분리 인식
- **지저분한 원본 파일**: 제목행/빈행이 표 위에 섞여 있어도 헤더 자동 탐지, 계정코드
  dtype 혼재("108.0" 등), 계정과목명 앞뒤 공백, 통화 코드 대소문자 혼용, 콤마 포함
  텍스트 숫자("15,000") 등을 자동 정규화
- **전표번호 중복(오채번)**: 같은 거래ID에 서로 다른 통화/거래처가 섞여 있으면
  데이터 무결성 경고를 띄우고, 명세서 기반 교차검증으로 실제 문제를 놓치지 않음
- **회사마다 다른 계정명/계정코드**: 건설업 현장별·국가별·본지점별로 계정이 쪼개진 경우도
  키워드 기반 계정매핑으로 대응 (아래 "계정매핑" 참고)
- **전표번호가 아예 없는 원본**: 차변=대변=0 규칙으로 전표 자동 채번 가능
- **100만 달러를 10번에 나눠 갚은 것처럼 발생-결제 1:1 매칭이 어려운 경우**: 거래ID 매칭
  없이도 거래처 단위 잔액 증감(롤포워드)만으로 총액 검증 가능

## 핵심 설계 포인트

- **적용환율 컬럼을 그대로 믿지 않음**: 분개장에 기재된 "적용환율" 라벨이 잘못돼 있어도 놓치지
  않도록, 실제 분개된 원화금액 ÷ 외화금액으로 **내재환율을 역산**해서 비교합니다.
- **개별 판정 + 합계 판정 이중 구조**: 괴리율이 작아도(예: 1.6%) 거래 규모가 크면 금액 차이가
  커질 수 있어, 개별 통과 건도 전부 합산해 중요성 기준(AMPT)으로 재검토합니다.
- **JPY 100엔 단위 고시 처리**: 수출입은행 API는 엔화를 100엔 단위로 고시합니다. 이 단위 환산을
  누락하는 실수(약 100배 과대/과소계상)를 탐지하는 로직을 포함합니다.
- **AI는 필요한 곳에만**: 결정론적으로 계산 가능한 부분(환율 재계산, 괴리율, 합계 검증)은
  전부 일반 로직으로 처리하고, **서식이 제각각인 증빙 문서를 읽어내는 부분에만** Claude Vision을
  사용했습니다.
- **매칭 없이도 되는 검증은 매칭에 의존하지 않음**: 발생-결제 거래를 1:1로 짝짓기 어려운 경우
  (분할상환 등)에도, 재고자산 롤포워드와 같은 원리(기초+발생-기말=소멸)로 거래처 단위 총액
  검증이 가능하도록 별도 체크를 두었습니다.

## 사용 방법

### 1. 환경 준비
```bash
pip install -r requirements.txt
```

### 2. 인증키 발급
- **한국수출입은행 환율 API**: [data.go.kr](https://www.data.go.kr/data/3068846/openapi.do)
  또는 [수출입은행 Open API](https://www.koreaexim.go.kr/ir/HPHKIR020M01) 에서 무료 발급
- **Anthropic API 키** (2단계 증빙 OCR용): [console.anthropic.com](https://console.anthropic.com)

### 3. 환경변수 설정
```bash
export EXIM_AUTH_KEY="발급받은_인증키"
export ANTHROPIC_API_KEY="발급받은_API_키"
export FX_AMPT="3000000"   # 허용가능 오류금액(Tolerable Misstatement) - 실제 감사건 중요성 금액으로 교체
```

### 4. 실행
```bash
python3 fx_verification_pipeline.py
```

기본적으로 `sample_data/` 폴더의 샘플 분개장·명세서·계정별원장을 대상으로 실행됩니다.
실제 회사 파일로 검증하려면 `COLUMN_MAP`(코드 상단)을 회사 파일의 실제 컬럼명에 맞게 수정하세요.

실행이 끝나면 콘솔 출력과 별도로 `검증결과.xlsx` 파일이 생성됩니다 (아래 "결과 엑셀 시트 구성" 참고).
이상치/부적정 행은 빨간색, 적정 행은 초록색으로 통째로 하이라이트되고 맨 위로 정렬되며,
차변금액/대변금액이 있는 시트에는 순액(차변-대변) 열이 자동으로 붙습니다.

### 5. 증빙 이미지로 2단계 검증
1단계에서 이상치로 플래그된 거래에 대해, `evidence/{거래ID}.png` (또는 `.jpg`, `.pdf`) 형태로
은행 외환거래확인서/SWIFT 통지서 등을 넣으면 자동으로 OCR 검증까지 진행됩니다.

### 6. 회사마다 다른 계정명 대응 (계정매핑)
회사의 계정과목 목록(시산표 등)을 `['회사계정코드', '회사계정명']` 형태로 준비해
`build_account_mapping()`에 넣으면, 키워드 기반으로 1차 자동분류하고 애매한 것만
"확인필요"로 표시합니다. 확인이 끝난 매핑은 `apply_account_mapping()`으로 분개장에
적용하면, 이후 파이프라인은 코드 수정 없이 그대로 동작합니다.

### 7. 전표번호가 없는 원본 (전표번호 자동채번)
`auto_assign_voucher_number()`를 쓰면 차변=대변=0이 되는 지점까지를 전표 1건으로
자동으로 묶어 전표번호를 부여합니다. 순수 회계 기계 규칙이라 사람 확인이 필요 없습니다.

## 결과 엑셀 시트 구성

| 시트 | 내용 |
|---|---|
| 안내 | 각 시트 설명 + 색상 범례 |
| 요약 | 합계 중요성(AMPT) 판정 결과 |
| A.표본전체상세 | 검증 대상이 된 분개 원본 라인 전체 + 거래별 최종 판정 (문제 있는 거래부터 정렬) |
| B.분석적검토 | 거래처별 연간 요약 + 월별 환율 추이 (분석적 절차 - 1차 검토용, 한 시트에 두 표) |
| C.외환차손익_상세 | 1단계 스크리닝 + 기준일 불일치 탐지 + 2단계 증빙검증을 거래 단위로 합친 표 |
| D.외화환산손익 | 기말 미결제 건 재평가 전수 검증 |
| E.완전성검증 | 계정별원장 대사 + 거래처별 롤포워드 검증 (한 시트에 두 표) |

## 샘플 데이터

`sample_data/`에 실제 한국수출입은행 API로 조회한 2025년 실제 환율을 기반으로 만든 8개 거래
샘플이 포함되어 있습니다. 정상 케이스와 아래와 같은 의도적 오류 케이스가 섞여 있습니다.

| 거래ID | 유형 | 내용 |
|---|---|---|
| TXN001, 005 | 정상 | 실제 환율 그대로 정확히 계상 (false positive 없음 확인용) |
| TXN002 | 오류 | 결제일이 아닌 전월말 환율을 잘못 적용 (개별 5% 기준은 통과 → 합계·기준일 탐지로 걸림) |
| TXN003 | 오류 | JPY 100엔 단위 나누기 누락으로 약 100배 과대계상 |
| TXN004 | 오류 | 환율 자체는 정확하지만 외환차익을 별도 계정으로 인식하지 않고 채권 계정에서 상계(계상 누락) |
| TXN006 | 정상 | 기말 미결제, 결산일 환율로 정확히 재평가 |
| TXN007 | 오류 | 기말 미결제 상태인데 재평가 분개 자체가 누락(완전성 이슈) |
| TXN008 | 오류 | 기말 재평가 시 당기말이 아닌 전기말 환율을 잘못 사용 |

각 케이스에 대한 상세 설명은 `sample_data/검증포인트_참고용.xlsx`에서 확인할 수 있습니다
(실제 회사 파일에는 없는, 검토용 정답지입니다).

## 강건성 테스트

`robustness_test/`에 실무에서 실제로 발생할 법한 지저분함과 복잡성(분할결제, 전표번호
중복 오채번, 텍스트로 입력된 숫자, 제목행이 섞인 원본 파일 등)을 의도적으로 담은
테스트 샘플과 생성 스크립트가 있습니다. 개발 중 이 샘플로 실제 로직을 검증하는 과정에서
분할결제 처리 버그 등을 발견하고 수정했습니다.

## 한계 및 향후 개선 방향

- 증빙-거래 매칭은 현재 거래ID 기준 파일명 매칭만 지원합니다. "날짜별 증빙 뭉치"에서 자동으로
  해당 거래를 찾아내는 기능은 아직 없습니다.
- OCR은 깔끔한 단일 이미지 기준으로 테스트했습니다. 흐릿한 스캔본, 여러 페이지 PDF 등에서는
  프롬프트 추가 조정이 필요할 수 있습니다.
- 결제일-환율기준일 불일치 탐지는 "흔한 오류 패턴"(전월말, 전영업일, 1~2주 전) 후보만 확인하며,
  전수 조합 탐색은 아닙니다 (API 호출 비용 절감을 위한 설계).
- **발생 전표와 결제 전표를 잇는 거래ID 매칭은 여전히 설계 단계입니다.** 전표번호 자동채번(차변=대변=0)은
  구현했지만, "이 발생 전표와 이 결제 전표가 같은 거래다"를 자동으로 판단하는 부분은 거래처+통화+
  금액+날짜 후보 추리기 + 사람 확인이 필요하다고 보고 있으며, 이 문서의 롤포워드 검증(D시트)이
  당장은 그 공백을 메우는 대안입니다.

## 기술 스택

Python · pandas · openpyxl · 한국수출입은행 Open API · Claude API (Vision, 증빙 OCR)
